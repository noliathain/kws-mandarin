1 · Accuracy vs capacity

At a fixed 15,000-step budget, token error rate falls steeply with the first 30× of parameters and then flattens. Read this as convergence rate, not a capacity ceiling — see §2: the same 1.65M model trained 4× longer reaches 0.255, beating the 6.2M model here. What this curve measures is how much accuracy each size buys per fixed budget.

2 · Validation trajectories

TER over training for each model. Bigger models start lower and stay lower; every curve is still gently descending at 15,000 steps, so these are lower bounds, not converged ceilings.

2b · Longer training beats more parameters

Every sweep curve was still descending at 15,000 steps, so we retrained the sweet-spot model (scale 20, 1.65M) for 60,000 steps — 4× the budget, same recipe. It converged at TER 0.2546, beating the 6.2M-parameter model trained at 15k.

Correction: an earlier version of this report claimed TER "saturates near 2M params" with a knee there. That was confounded — at 15k steps none of the six models had converged, so the flattening measured convergence rate at a fixed budget, not a capacity ceiling. A 1.65M model given enough training beats a 6.2M model that was not.

Still open: what scale 30/40 reach at 60k steps. They may go lower still, so this does not establish 1.65M as optimal — only that the earlier saturation claim does not hold as stated.

Implication: the 60k run saw 256 epochs of the 150 h corpus and plateaued only at the very end, with no overfitting. With training time no longer binding at this size, data volume is now the most promising untested axis.

3 · The cost of capacity

Bigger models train slower per step (all six ran at global batch 512 on 2×H100). The accuracy-vs-throughput trade sets where to stop for a given budget.

Reading this: TER is the reliable signal — measured over the full dev set, it falls monotonically with size at this fixed budget (the apparent saturation past scale 30 is budget-driven, not a capacity limit). FRR is measured on 500 utterances × 10 keywords, so single-scale wobbles (e.g. scale 30's 0.25) are within noise; the trend, not the point, is what to trust.

4 · Noise robustness (scale 20)

The sweet-spot model degrades gracefully as noise rises — no cliff. False-reject rate at 1 alarm/hour roughly triples from clean to 0 dB, but the system keeps working through café-level noise (~5 dB). This is the payoff of training with additive noise, finally measured.

Caveat: the noise here is MUSAN — the same corpus the model trained on — so this is robustness to the training distribution, a floor rather than held-out generalization. A truly unseen source (WHAM!/DEMAND) is the honest follow-up; the eval script already accepts one.

5 · How the pipeline got fast enough to run this

The sweep was only affordable because the training step went from data-bound to compute-bound first. Six full runs at the original 1.4 it/s would have taken ~18 h of wall clock; at 7.5 it/s (scale 3) the whole sweep fit in an afternoon across a few boxes.

baseline 1.4 it/s→ parallel shard reads +I/O→ cache corpus in RAM→ GPU augmentation→ background prefetch→ tokenizer cache→ scale 3 7.5 it/s

5.4× faster, GPU utilization 36% → ~98%. The step became compute-bound, so the remaining per-step time now scales with the model — which is exactly what makes the throughput column above a clean read of model cost.

All points: identical recipe (lr 3e-3, warmup 1000, cosine decay, bf16, EMA with warmup, RIR+MUSAN+speed augmentation on GPU), global batch 512 via micro-batch 64 × accum 4 × 2 GPUs, so BatchNorm statistics match across scales. Only model.scale varies.
